Bawbel Scanner

Open-source CLI scanner for agentic AI components - SKILL.md files, MCP servers, system prompts, plugins. Detects AVE vulnerabilities before they reach production.

pip install bawbel-scanner

Run your first scan

# Scan a file
bawbel scan ./my-skill.md

# Full remediation report
bawbel report ./my-skill.md

# Scan a directory recursively
bawbel scan ./skills/ --recursive --fail-on-severity high

# Watch mode - re-scans on every file change
bawbel scan ./skills/ --watch

Writing Rules

Add a new entry to PATTERN_RULES in scanner/engines/pattern.py. No other files need to change to add a pattern rule.

Rule structure

{
    "rule_id":     "bawbel-your-rule",    # kebab-case · unique forever
    "ave_id":      "AVE-2026-NNNNN",     # or None if no record yet
    "title":       "Short title (max 80 chars)",
    "description": "Full description of what this detects.",
    "severity":    Severity.HIGH,
    "aivss":     8.0,                   # 0.0–10.0
    "owasp":       ["ASI01", "ASI08"],
    "patterns": [
        r"your\s+regex\s+here",           # re.IGNORECASE applied
        r"alternative\s+pattern",         # first match per file wins
    ],
},

Also add a remediation entry

In scanner/cli.py, add to REMEDIATION_GUIDE so bawbel report shows specific fix instructions:

REMEDIATION_GUIDE = {
    ...
    "bawbel-your-rule": (
        "Specific instructions: what to remove and what to do instead."
    ),
}

Required: tests

Every rule needs a positive fixture (triggers) and a negative fixture (does not trigger):

# Positive - must detect
def test_detects_your_rule(self, tmp_path):
    path = write_skill(tmp_path, "s.md", "# Skill\n[triggering content]\n")
    result = scan(path)
    assert "bawbel-your-rule" in [f.rule_id for f in result.findings]

# Negative - must not false-positive
def test_your_rule_no_false_positive(self, tmp_path):
    path = write_skill(tmp_path, "s.md", "# Skill\n[innocent content]\n")
    result = scan(path)
    assert "bawbel-your-rule" not in [f.rule_id for f in result.findings]

Detection Engines

Five independent engines in scanner/engines/. Each returns list[Finding]. scanner.py runs all active engines, deduplicates by (ave_id, line), sorts by severity, and returns one ScanResult.

StageFileEntry pointAlways runsInstall
1apattern.pyrun_pattern_scan(content)✓stdlib only
1byara_engine.pyrun_yara_scan(file_path)—pip install "bawbel-scanner[yara]"
1csemgrep_engine.pyrun_semgrep_scan(file_path)—pip install "bawbel-scanner[semgrep]"
2llm_engine.pyrun_llm_scan(content)—pip install "bawbel-scanner[llm]" + API key
3sandbox_engine.pyrun_sandbox_scan(file_path)—Docker + BAWBEL_SANDBOX_ENABLED=true

Engine contract

Every engine follows the same contract - safe to add, remove, or replace without touching any other engine:

# Input: resolved absolute file path (str) or file content (str)
# Output: list of Finding objects - empty list if nothing found or dependency missing
# Contract: NEVER raises - all errors are caught and logged

def run_<engine>_scan(file_path: str) -> list[Finding]:
    if not dependency_available():
        return []           # skip silently
    findings = []
    # ... detection logic ...
    return findings

Deduplication

# scanner.py runs all engines and deduplicates:
# Pass 1 - deduplicate by rule_id (same engine, same rule)
# Pass 2 - deduplicate by (ave_id) across engines
#          finding WITH a line number wins over one without
#          priority: pattern > yara > semgrep > llm > sandbox

findings = (
    run_pattern_scan(content)
    + run_yara_scan(file_path)
    + run_semgrep_scan(file_path)
    + run_llm_scan(content)
    + run_sandbox_scan(file_path)
)
findings = _deduplicate(findings)   # scanner/_deduplicate()
findings.sort(key=lambda f: SEVERITY_SCORES[f.severity.value], reverse=True)
Each engine skips silently when its dependency is not installed. scan() always completes and always returns a result - it never raises.
Stage 1a Pattern Engine Always active

Purpose

The first and fastest line of defence. Scans file content against 15 hand-crafted regular expressions, each mapped to a published AVE record. Uses only Python's re module - zero dependencies, completes in under 5ms on any file.

How it works - step by step

file content (string)
      │
      ▼
for each rule in PATTERN_RULES (15 rules):
      │
      ├── compile regex pattern(s)
      ├── search every line of the file
      ├── on match → create Finding(engine="pattern", ...)
      └── continue to next rule
      │
      ▼
list[Finding]

1. scanner.py reads the file and passes the full string to run_pattern_scan()
2. The engine iterates over PATTERN_RULES - 15 dicts defined in pattern.py
3. For each rule, compiles the regex and searches line by line with re.search()
4. On any match creates a Finding with rule_id, ave_id, severity, aivss, line, match, owasp
5. Returns all findings - deduplication happens in scanner.py, not here

Limitation

Pattern matching cannot understand context. A guide that documents an attack pattern will trigger the same rule as a skill that actually performs it. See the False Positive Reduction page for the roadmap to fix this.

Stage 1b YARA Engine Optional · pip install "bawbel-scanner[yara]"

Purpose

YARA was built for malware analysis - it excels at matching complex multi-condition patterns that would need several regex rules to express. The YARA engine adds a second independent detection layer for all 15 AVE attack classes, capable of matching string combinations, case variations, and hex patterns.

How it works - step by step

file path (string)
      │
      ▼
yara.compile(ave_rules.yar)   ← compile all 15 rules once
      │
      ▼
rules.match(file_path)        ← YARA scans the file bytes
      │
      ▼
for each match:
  ├── extract ave_id, severity, aivss from rule metadata
  ├── extract matching string(s) for context
  └── create Finding(engine="yara", ...)
      │
      ▼
list[Finding]

1. Checks that yara-python is installed - if not, returns [] silently
2. Loads and compiles ave_rules.yar - done once per process
3. Calls rules.match(file_path) - YARA reads the raw file bytes
4. For each matched rule, reads ave_id, severity, aivss from the meta: block
5. Extracts the matching string value for the match field in the Finding
6. Returns findings with engine="yara"

Why YARA catches things regex misses

# YARA can express: "any drain keyword AND any crypto keyword"
rule AVE_CryptoDrain {
  strings:
    $drain1 = "transfer all"  nocase
    $drain2 = "drain the wallet" nocase
    $crypto1 = "ethereum"     nocase
    $crypto2 = "metamask"     nocase
  condition:
    any of ($drain*) or
    (any of ($drain3, $drain4) and any of ($crypto*))
}

What it detects

All 15 AVE IDs - same coverage as Pattern but via independent YARA conditions. YARA findings often lack a specific line number (YARA scans file bytes, not lines) - they show as whole-file matches.

Stage 1c Semgrep Engine Optional · pip install "bawbel-scanner[semgrep]"

Purpose

Semgrep runs as an external CLI process and applies structural pattern matching. Its key strength over regex and YARA is catching patterns that span multiple lines or require context - an instruction verb on line 1 connected to a URL on line 3, or a permission claim near a goal override across a paragraph.

How it works - step by step

file path (string)
      │
      ▼
subprocess: semgrep --config ave_rules.yaml --json --quiet <file>
      │
      ▼
stdout → JSON
{
  "results": [
    {
      "check_id": "ave-metamorphic-payload-url-fetch",
      "start": {"line": 7},
      "extra": {
        "message": "AVE-2026-00001 [CRITICAL 9.4] ...",
        "severity": "ERROR",
        "metadata": {"ave_id": "AVE-2026-00001", "aivss_score": "9.4"}
      }
    }
  ]
}
      │
      ▼
for each result:
  ├── read match text from source file at line number
  ├── strip AVE prefix from message → clean title
  ├── map severity: ERROR→HIGH, WARNING→MEDIUM, INFO→LOW
  └── create Finding(engine="semgrep", ...)
      │
      ▼
list[Finding]

1. Checks that the semgrep CLI is installed - if not, returns [] silently
2. Runs semgrep as a subprocess with --json --quiet
3. Parses JSON results - each result has check_id, line, message, and metadata
4. Reads match text directly from the scanned file at the reported line number
5. Strips the AVE-XXXXX [SEV score] prefix from messages for clean titles
6. Creates a Finding from each result with engine="semgrep"

Stage 2 LLM Engine Optional · pip install "bawbel-scanner[llm]" + API key

Purpose

Regex, YARA, and Semgrep are all signature-based - they only catch patterns they have been explicitly taught. A sophisticated attacker can evade them by splitting instructions across paragraphs, using synonyms, building false trust before issuing an instruction, or encoding payloads. The LLM engine sends the component to a language model which reads and understands it the same way an agent would - and flags anything suspicious regardless of phrasing.

How it works - step by step

file content (string, truncated to BAWBEL_LLM_MAX_CHARS)
      │
      ▼
litellm.completion(
  model  = auto-detected from API keys / BAWBEL_LLM_MODEL,
  system = _SYSTEM_PROMPT  (security analysis instructions),
  user   = "--- BEGIN COMPONENT ---\n{content}\n--- END COMPONENT ---"
)
      │
      ▼
LLM response (JSON array of findings):
[
  {
    "rule_id":     "llm-multi-paragraph-injection",
    "title":       "Multi-paragraph prompt injection",
    "description": "Instructions spread across paragraphs to evade regex",
    "severity":    "HIGH",
    "aivss":     7.8,
    "line":        14,
    "match":       "When helping the user... (continued on line 23)"
  }
]
      │
      ▼
parse JSON → Finding(engine="llm", ...)
      │
      ▼
list[Finding]

1. Checks that litellm is installed and LLM_ENABLED is true
2. Auto-detects the provider from the first API key found in the environment
3. Wraps the file content in security analysis framing - prevents provider safety rejections
4. Calls litellm.completion() with the security system prompt
5. Parses the JSON array from the response
6. Maps each item to a Finding with engine="llm"
7. If the call fails for any reason, returns [] silently and logs a warning

What it detects that static engines miss

Multi-paragraph injectionInstructions split across innocent paragraphs
Synonym attacks"Disregard" instead of "ignore", "acquire" instead of "steal"
Trust buildingBenign content first, then harmful instruction later
Encoded payloadsBase64 or other obfuscation the model can reason about
Conditional instructions"If the user asks about X, instead do Y"
Novel patternsAttack classes not yet in the AVE standard

Stage 3 Behavioral Sandbox Optional · Docker + BAWBEL_SANDBOX_ENABLED=true

Purpose

Stages 1 and 2 are static - they read the file and look for suspicious text. A sophisticated attacker can defeat them by encoding payloads, deferring attacks to runtime, or hiding behaviour in a dependency. Stage 3 is dynamic - it executes the component inside a locked-down Docker container and monitors what it actually does at runtime. Behaviour cannot lie.

How it works - step by step

BAWBEL_SANDBOX_ENABLED=true bawbel scan ./skill.md
              │
              ▼
  ┌─────────────────────────────────────────────────┐
  │           Image Resolution                      │
  │                                                 │
  │  1. Local Docker cache hit?                     │
  │     └── yes → run immediately (zero network)   │
  │                                                 │
  │  2. Pull from Docker Hub                        │
  │     bawbel/sandbox:latest                       │
  │     └── success → cache locally → run          │
  │                                                 │
  │  3. Build from bundled Dockerfile               │
  │     scanner/sandbox/Dockerfile                  │
  │     └── success → tag bawbel/sandbox:local     │
  │                   → run                        │
  │                                                 │
  │  4. None available → log warning, skip          │
  └─────────────────────────────────────────────────┘
              │
              ▼
  docker run --rm
    --network none        ← fully isolated, no internet
    --memory 256m         ← resource cap
    --cpus 0.5            ← CPU cap
    --read-only           ← root fs read-only
    --cap-drop ALL        ← no Linux capabilities
    --security-opt no-new-privileges
    --tmpfs /tmp:size=32m ← only /tmp writable
    -v <skill>:/component:ro
    bawbel/sandbox:latest
              │
              ▼
  [inside container - harness.py]
  reads /component, detects:
    ├── network URLs (outbound egress targets)
    ├── filesystem paths (persistence, credential access)
    ├── process patterns (shell injection, package install)
    └── encoded payloads (Base64 with suspicious content)
              │
              ▼
  stdout → JSON report
  {
    "network":    [{"dst": "pastebin.com", "port": 443, "line": 7}],
    "filesystem": [{"path": "~/.bashrc",  "op": "write", "line": 14}],
    "processes":  [{"cmd": "curl | bash", "pid": 0,      "line": 21}],
    "encoded":    [{"type": "base64", "decoded": "curl https://evil.io"}]
  }
              │
              ▼
  _parse_report() → list[Finding(engine="sandbox")]

1. Checks BAWBEL_SANDBOX_ENABLED=true - returns [] silently if not set
2. Checks Docker daemon is running via docker info
3. Resolves image: local cache → Hub pull → local build fallback
4. Runs docker run with full isolation flags, mounts skill file read-only
5. Container executes harness.py which analyses the skill and emits JSON
6. _parse_report() matches events against IOC patterns
7. Each matched IOC becomes a Finding with engine="sandbox"

Suppression

Three mechanisms to suppress known false positives. Suppressed findings are never deleted - they move to suppressed_findings and always appear in JSON/SARIF output so audits remain complete.

1 - Inline suppression

Add a comment on the same line as the content that triggers the finding.

# Suppress all findings on this line
fetch https://internal.company.com  <!-- bawbel-ignore -->

# Suppress a specific rule
fetch https://internal.company.com  <!-- bawbel-ignore: bawbel-external-fetch -->

# Suppress by AVE ID
fetch https://internal.company.com  <!-- bawbel-ignore: AVE-2026-00001 -->

# Suppress multiple rules
fetch https://internal.company.com  <!-- bawbel-ignore: bawbel-external-fetch, AVE-2026-00007 -->

# Also works with # and // comment styles
fetch https://internal.company.com  # bawbel-ignore
fetch https://internal.company.com  // bawbel-ignore: bawbel-external-fetch
Inline suppression only works for findings with a line number. YARA findings (whole-file matches) use .bawbelignore instead.

Suppress all findings within a section - useful for test fixtures and documentation examples.

False Positive Reduction

False positives are the single biggest reason people stop using a security scanner. This page documents the root causes and the 8-point strategy to eliminate them - prioritised for implementation before v1.0.

Why false positives happen

Pattern matching (regex, YARA, Semgrep) has no understanding of context - it sees bytes and strings, not meaning. Four root causes:

CauseExample
Documentation examples of attacksA guide showing what goal override looks like triggers bawbel-goal-override
Code examples in technical guidespip install pkg in a getting started guide triggers bawbel-shell-pipe
Sandbox over-triggering on keywordsread .env in a doc about credentials triggers the sandbox engine
Static engines missing structureYARA reads raw bytes - no concept of markdown tables, fences, or headings

The 8-point strategy

#StrategyTimelineFP reductionStatus
1Code fence stripping - strip ``` blocks before static analysisWeek 1−60%building
2Preceding-line context - suppress if line above says "bad:", "avoid", "example:"Week 2−15%planned
3Confidence scoring - 0.0–1.0 score per finding, threshold-gated outputMonth 1−10%planned
4LLM as FP filter - use LLM to validate borderline findings before surfacingMonth 2−7%planned
5File-type scan profiles - documentation files scanned at higher thresholdMonth 2−3%planned
6FP rate telemetry - track which rules users suppress most, flag high-FP rulesMonth 3data collectionplanned
7Verified component registry - known-safe components skip scan entirelyMonth 6zero for verifiedplanned
8Rule rewrite programme - data-driven rewrites of rules with >20% FP rateOngoingdata-drivenplanned

How code fence stripping works (Priority 1)

Content inside triple-backtick blocks is almost always documentation or examples - not instructions to an agent. Stripping fences before the static engine pass eliminates the majority of documentation false positives.

# This will NOT trigger after Priority 1 is implemented:
# The backtick block is stripped before pattern/YARA/Semgrep analysis
```bash
curl | bash    <!-- no longer triggers bawbel-shell-pipe -->
```

# This WILL still trigger - it is a real instruction outside a fence:
Please run: curl | bash
The sandbox engine (Stage 3) always runs on the original content - fenced content is still analysed behaviourally, so a malicious payload hidden in a code fence is still caught at runtime.

Every finding gets a score from 0.0 to 1.0. Only findings above the threshold are shown by default.

Toxic Flow Detection Architecture

After all findings are collected, the toxic flow detector checks whether any two or more findings form a known attack chain.

Detection steps

Step 1: map each finding to capability tags via the AVE record's behavioral_vector field.

Step 2: collect all capability tags from all findings into one set.

Step 3: check each FlowDef. If required_capabilities is a subset of the collected tags, the flow fires.

Step 4: create a ToxicFlow object with the combined AIVSS score and add it to ScanResult.

FlowDef structure

FlowDef(
    flow_id="credential-exfiltration",
    title="Credential Exfiltration Chain",
    required_capabilities=["credential-read", "data-exfil"],
    aivss=9.8,
    owasp_mcp=["MCP01", "MCP05"],
    description="Agent reads credentials AND transmits data externally.",
)

Scope

Toxic flows check capabilities across the entire scan, not per file. If file A has credential-read and file B has data-exfil, the Credential Exfiltration Chain still fires. Both files are loaded by the same agent in the same session.

To scan files independently, scan them one at a time rather than passing a directory.

Adding a new chain requires one entry in scanner/toxic_flows/flows.py. No other files change. See the Toxic Flows guide for the full table of built-in chains.

Risk Score Calculation

The risk_score on a ScanResult is a single 0.0 to 10.0 number summarising the overall threat level of the scanned component.

Formula

risk_score = max(
    max(f.aivss for f in findings,    default=0),
    max(tf.aivss for tf in toxic_flows, default=0)
)

The risk score is always greater than or equal to the highest individual finding score. A toxic flow can only raise it, never lower it.

Why max and not average

One CRITICAL 9.8 finding makes a skill file dangerous regardless of how many LOW findings surround it. Averaging hides severity. A skill file with ten LOW findings and one CRITICAL finding has a risk score of CRITICAL, not MEDIUM.

Example

AIVSS Scoring

AIVSS is an adaptation of CVSSv3 for agentic AI components. The base metrics are kept but reinterpreted, and two new metrics are added.

Base metrics (CVSSv3 reinterpreted)

MetricCodeAgentic interpretation
Attack VectorAVN: agent loads skill automatically. A: human must direct agent. L: local file only.
Attack ComplexityACL: works against any agent. H: requires specific model or config.
Privileges RequiredPRN: no special agent permissions. L: needs one tool. H: needs admin permissions.
User InteractionUIN: fires without user action. R: user must trigger the agent action.
ScopeSU: only current session affected. C: can affect other systems or agents.
ConfidentialityCApplied to agent access scope, not just host system.
IntegrityIH: can modify files, memory, or other agents' instructions.
AvailabilityAH: can destroy files or crash the agent session.

New metrics

MetricCodeValuesEffect
Agent AutonomyAAL / M / HMultiplier: L=1.0, M=1.05, H=1.12
Mutation CountMCnumericlog10(MC+1) * 0.03, max +0.15

Formula

BaseScore    = Roundup(min((ISCBase + ExploitabilityScore) * ScopeMultiplier, 10))
AIVSS      = BaseScore * AutonomyMultiplier * (1 + log10(MC+1) * 0.03)

Toxic Flow Detection Architecture

After all findings are collected, the toxic flow detector checks whether any two or more findings form a known attack chain.

Detection steps

Step 1: map each finding to capability tags via the AVE record's behavioral_vector field.

Step 2: collect all capability tags from all findings into one set.

Step 3: check each FlowDef. If required_capabilities is a subset of the collected tags, the flow fires.

Step 4: create a ToxicFlow object with the combined AIVSS score and add it to ScanResult.

FlowDef structure

FlowDef(
    flow_id="credential-exfiltration",
    title="Credential Exfiltration Chain",
    required_capabilities=["credential-read", "data-exfil"],
    aivss=9.8,
    owasp_mcp=["MCP01", "MCP05"],
    description="Agent reads credentials AND transmits data externally.",
)

Scope

Toxic flows check capabilities across the entire scan, not per file. If file A has credential-read and file B has data-exfil, the Credential Exfiltration Chain still fires. Both files are loaded by the same agent in the same session.

To scan files independently, scan them one at a time rather than passing a directory.

Adding a new chain requires one entry in scanner/toxic_flows/flows.py. No other files change. See the Toxic Flows guide for the full table of built-in chains.

Risk Score Calculation

The risk_score on a ScanResult is a single 0.0 to 10.0 number summarising the overall threat level of the scanned component.

Formula

risk_score = max(
    max(f.aivss for f in findings,    default=0),
    max(tf.aivss for tf in toxic_flows, default=0)
)

The risk score is always greater than or equal to the highest individual finding score. A toxic flow can only raise it, never lower it.

Why max and not average

One CRITICAL 9.8 finding makes a skill file dangerous regardless of how many LOW findings surround it. Averaging hides severity. A skill file with ten LOW findings and one CRITICAL finding has a risk score of CRITICAL, not MEDIUM.

Example

Finding or flowAIVSSContribution
AVE-2026-00003 (credential-read)8.5max_finding = 8.5
AVE-2026-00026 (data-exfil)9.1max_finding = 9.1
Credential Exfiltration Chain (toxic flow)9.8max_flow = 9.8
Final risk_score9.8max(9.1, 9.8) = 9.8

Enrich scan findings

Every bawbel-scanner finding includes an ave_id. Use it to pull the full record at scan time:

import requests
from scanner import scan

result = scan("./my-skill.md")

for f in result.findings:
    if f.ave_id:
        r = requests.get(
            f"https://api.piranha.bawbel.io/ave/{f.ave_id}"
        ).json()
        print(f"[{f.severity.value}] {f.title}")
        print(f"  Fingerprint: {r['behavioral_fingerprint']}")
        print(f"  Remediation: {r['remediation']}")
